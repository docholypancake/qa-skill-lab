# Module 02 — API Testing with Postman

## What's tested

All 14 test cases from Module 1 executed against the [Restful-Booker](https://restful-booker.herokuapp.com/apidoc/index.html) public API using Postman with `pm.test()` assertions and Newman CLI.

Endpoints covered: `GET /ping`, `POST /auth`, `POST /booking`, `GET /booking/{id}`, `GET /booking`, `PUT /booking/{id}`, `PATCH /booking/{id}`, `DELETE /booking/{id}`.

## Files

| File | Description |
|------|-------------|
| `booker-collection.json` | Postman collection v2.1 — all 14 requests with assertions |
| `restful-booker-environment.json` | Environment template — `base_url`, `token`, `booking_id` |

The `postman/` folder at repo root contains the same collection in Postman's git-native YAML format (auto-generated by Postman desktop).

## How to run

### Postman (GUI)

1. Import `booker-collection.json` and `restful-booker-environment.json` into Postman
2. Select the `restful-booker` environment
3. Open Collection Runner — run requests in this order:

```
TC-RB-003 → TC-RB-002 → TC-RB-004 → TC-RB-001 → TC-RB-005 →
TC-RB-006 → TC-RB-007 → TC-RB-008 → TC-RB-009 → TC-RB-010 →
TC-RB-012 → TC-RB-013 → TC-RB-014 → TC-RB-011
```

> Order matters: TC-RB-001 creates the booking (sets `booking_id`), TC-RB-004 fetches the auth token, TC-RB-011 deletes the booking and must run last.

### Newman (CLI)

```bash
npm install -g newman

newman run 02-api-postman/booker-collection.json \
  --environment 02-api-postman/restful-booker-environment.json
```

## Intentional failures

These two tests are designed to fail. They document known Restful-Booker defects:

**TC-RB-013** — `POST /booking` with `totalprice: -1` returns **200 OK** (should be 400). API accepts negative prices with no validation.

**TC-RB-014** — `POST /booking` with missing `firstname` returns **500 Internal Server Error** (should be 400). Missing required field causes a server crash — severity High.

## QA skills demonstrated

- Postman environment variables (`{{base_url}}`, `{{token}}`, `{{booking_id}}`) to avoid hardcoded values
- `pm.environment.set()` to chain data between requests (bookingid, token)
- Assertion depth: status code + response shape + business rule (never status alone)
- Dependency ordering: DELETE last to preserve test data for preceding cases
- Intentional failure cases as living defect documentation
- Newman CLI for headless execution (CI-ready)
